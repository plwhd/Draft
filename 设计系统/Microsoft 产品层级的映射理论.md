# Microsoft 产品层级的映射理论

> 本文系统性整理微软在复杂产品中对**结构、层级与信息组织**的底层理解方式。  
> 核心结论：**产品层级不是 UI 排序问题，而是认知映射问题。**

---

## 一、产品层级的根本定义：映射，而非表现

在微软的设计体系中，产品层级并不是由 UI 视觉元素直接决定的。  
微软真正关心的，是三个不同“世界”之间是否形成了稳定、可理解的映射关系。

### 1. 层级的定义

> **层级 = 三个世界之间的稳定映射关系**

### 2. 三个世界

- **用户的心理模型（Mental Model）**：用户如何理解自己在做什么  
- **系统的概念模型（Conceptual Model）**：系统中真实存在的对象及其关系  
- **界面的感知模型（Perceptual Model）**：UI 如何呈现这些对象与关系  

### 3. 微软的设计目标

> 微软做的不是“好看的 UI”，而是让这三者尽可能 **同构（Isomorphic）**。

---

## 二、第一层：心理模型 —— 结构的来源

### 1️⃣ 心理模型决定“什么该在上层”

微软的基本假设是：

> **层级不是由信息重要性决定的，而是由“用户在想什么”决定的。**

### 2. 用户心理模型中的三类对象

- **目标（Goal）**：我想完成什么  
- **对象（Object）**：我在操作什么  
- **操作（Action）**：我能对它做什么  

### 3. 最基础的层级骨架（原样呈现）

目标（Goal)  
↓  
对象（Object)  
↓  
操作（Action)

> 在微软产品里：**目标 > 对象 > 操作**  
> 这构成最基础的层级骨架。

### 4. 示例

- 文件（Object）  
- 文件夹（Object 的容器）  
- 打开 / 编辑 / 分享（Action）

---

## 三、第二层：概念模型 —— 层级的“真实结构”

> 这是微软最强、但很少被显式讲清的一层。

### 2️⃣ 微软把产品当成一个「对象系统」

> **产品 = 一组对象 + 对象之间的关系**

### 常见关系类型（映射的核心）

| 关系类型 | 含义 |
|--------|------|
| 包含（Containment） | A 包含 B |
| 从属（Ownership） | B 属于 A |
| 引用（Reference） | A 指向 B |
| 状态（State） | A 的某种状态 |
| 行为（Capability） | A 能做什么 |

> **层级本质上就是：关系的可视化。**

### 3️⃣ 映射关系如何决定层级？

#### 包含关系 → 空间嵌套

- Folder → File  
- Section → Item  
- Project → Task  

#### 从属关系 → 导航层级

- Account → Settings  
- Workspace → Members  

#### 状态关系 → 层级切换

- 草稿 / 发布  
- 启用 / 禁用  

#### 行为关系 → 操作层级

- Primary action  
- Secondary action  
- Destructive action  

> 微软的 UI 只是把这些关系翻译成视觉和交互语言。

---

## 四、第三层：感知模型 —— 层级如何被“看到”

现在才轮到设计师熟悉的东西，但要注意：

> **这里不是在“定义层级”，而是在“呈现层级”。**

### 4️⃣ 感知层级 ≠ 真实层级

微软刻意避免：

- 所有关系都同时显性化  
- 所有结构都可见  

他们遵循一个认知原则：

> **用户一次只能理解有限层级**

所以：

- 有些关系通过 **空间**  
- 有些通过 **时间（交互）**  
- 有些通过 **隐含规则**

---

## 五、映射理论的核心：结构 ≠ 层级

这是一个关键分界点。

### 结构（Structure）

- 系统真实存在的关系网络  
- 多对多  
- 非线性  

### 层级（Hierarchy）

- 在特定情境下  
- 对结构的投影（projection）

> 微软不是做“树结构产品”，而是：在不同任务下，选择不同的层级视角。

---

## 六、为什么微软产品“感觉有层级，但不死板”

因为他们遵循这条规则：

> **同一个对象，在不同映射下，可以处于不同层级。**

举例（真实 Fluent 思路）：

文件在：

- 导航中是一级对象  
- 搜索结果中是平级对象  
- 编辑状态中是上下文中心  

> **层级是上下文相关的。**

---

## 七、一个真正“微软式”的层级定义流程

如果你要从 0 定义一个产品层级，微软的隐性流程大概是：

1. **列出对象（不是页面）**：用户关心的是什么“东西”  
2. **定义对象关系**：包含 / 从属 / 引用 / 状态 / 行为  
3. **确定当前任务视角**：用户现在在解决什么问题  
4. **选择一种关系作为主映射**：导航？空间？流程？  
5. **把其他关系降级或延迟**：通过交互再暴露  

> **层级不是设计出来的，是被“选出来的”。**

---

## 八、一句话总结（非常底层）

> **产品层级不是信息排序，而是对“对象关系网络”的认知投影。**

> **映射关系 = 心理模型 × 概念模型 × 当前任务**
